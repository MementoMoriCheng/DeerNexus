# PR-025A — Enforce `org_id NOT NULL` + `threads_meta` 复合唯一约束

## Why

Track B 的 Expand（PR-021，可空 `org_id`）、Backfill（PR-023，存量行已全部回填默认 Org）、Repository Org Scope（PR-024，仓储写入盖戳 / 读取硬过滤 / fail-closed）均已交付。至此**应用层**已保证：每条新写入都带 `org_id`、缺失 tenant 上下文 fail-closed、跨 Org 行被数据库层排除。但**数据库层**仍只是 `nullable=True`——若绕过应用（直连 DB 的脚本、误操作、旧版本应用连接）即可写入 NULL `org_id`，破坏"空 `org_id=0`"这一多 Org 启用前置（production-runbook §5.2 把该状态判为 FAIL）。

PR-025A 是 data-model §13.3 **Enforce** 阶段的 schema 部分：把 4 张 Run-lifecycle 资源表（`threads_meta`/`runs`/`run_events`/`feedback`）的 `org_id` 收紧为 `NOT NULL`，并按 §7.1 增加 `UNIQUE(org_id, thread_id)` 声明性约束。这是把"应用承诺"升级为"数据库强制"的不可逆（生产语义）迁移——本 PR 后，任何 NULL `org_id` 写入在 DB 层即被拒。

**范围决定（用户确认）**：本次只交付 PR-025A。`pr-split-guide §7` 明确 `不得把 A–D 合成一次不可回滚发布`，`ci-cd §10.3` 把 B（Feature Flag 启用）硬性门禁在 A 上线且双 Org 矩阵全绿、空 `org_id=0` 之后。B/C/D 全部留后续。

## Architecture / approach

### 1. `safe_set_column_nullable` helper（`migrations/_helpers.py`）

镜像既有 `safe_*` 家族（`safe_add_column`/`safe_drop_column`/`safe_create_index`/`safe_drop_index`/`safe_create_table`）。签名：

```python
def safe_set_column_nullable(
    table: str, column_name: str, *, nullable: bool, existing_type: sa.TypeEngine
) -> None:
    """batch-alter 一列的 nullability；已在目标态则 no-op（+ 类型 drift warning）。"""
```

逻辑：`_inspector()` → 表不存在则 return；取现有列；若 `existing["nullable"] == nullable` 则 no-op（顺带 `_check_column_drift` 风格的类型 warning）；否则 `with op.batch_alter_table(table) as batch: batch.alter_column(column_name, existing_type=existing_type, nullable=nullable)`。batch 模式自动 reflect 并重新应用命名 FK `fk_<table>_org_id`（0005 `_org_id_column` docstring 行 93-100 已证实此行为），故 FK 无损。

### 2. 迁移 `0006_enforce_org_not_null`

`revision="0006_enforce_org_not_null"`，`down_revision="0005_resource_org_id"`。

```python
_RESOURCE_TABLES = ("threads_meta", "runs", "run_events", "feedback")

def upgrade():
    for table in _RESOURCE_TABLES:
        safe_set_column_nullable(table, "org_id", nullable=False, existing_type=sa.String(36))
    safe_create_index("uq_threads_meta_org_thread", "threads_meta", ["org_id", "thread_id"], unique=True)

def downgrade():
    safe_drop_index("uq_threads_meta_org_thread", "threads_meta")
    for table in _RESOURCE_TABLES:
        safe_set_column_nullable(table, "org_id", nullable=True, existing_type=sa.String(36))
```

docstring 要点：标"不可逆（生产语义）"——Enforce 后生产 DB 无法容忍 NULL `org_id`；downgrade 在应用层仍 stamp org_id 的前提下安全（PR-024 保证新写必带 org_id，fail-closed 不产生新 NULL）；残留 NULL 会让 ALTER fail-loud（正确）。

## Tenant / Security
- DB 层强制 `org_id` 非空：堵死绕过应用写入 NULL 的路径（直连脚本、误操作、旧应用），呼应 §5.2 rule 6 fail-closed 的数据侧兜底。
- `UNIQUE(org_id, thread_id)` 当前为声明性（`thread_id` 已全局 PK，故恒满足），确立 §1.9 org 前缀唯一范式，为后续 org-scoped 业务键铺路。
- 不动认证/授权/过滤逻辑（PR-024 已就位）。

- 无新列、无 backfill（PR-023 已让存量全非空）。
- NOT NULL ALTER 在有残留 NULL 时被 DB 拒绝（fail-loud）；零数据丢失前提：空 `org_id=0`（PR-023 保证）。

## 风险与缓解
- **不可逆（生产语义）**：Enforce 后生产 DB 不容忍 NULL `org_id`。缓解：PR-023 backfill 已覆盖、PR-024 新写必 stamp、迁移 fail-loud；doctor（PR-025C）后续补友好预检。
- **SQLite batch ALTER 锁/重建开销**：大表重建有锁开销。缓解：经 bootstrap 既有锁串行；4 表为 Run-lifecycle 元数据，体量可控；runbook 禁"未验证索引锁进入生产"——本 PR 无新索引重建（唯一约束小），风险低。
- **parity 测试漂移**：模型与迁移必须锁步。缓解：4 模型 nullable + threads_meta UniqueConstraint 与迁移同 PR 改；parity 测试即守门。
